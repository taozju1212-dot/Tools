## 1. 软件概述

### 1.1 功能定位

FA120 日志解析软件用于解析和分析 **FA120 全自动荧光免疫分析仪** 产生的运行日志，帮助工程师进行：

- **故障诊断**：快速定位报警事件与异常动作
- **时序分析**：可视化各电机/组件的运动时序，识别超时动作
- **完整性检查**：对比标准动作序列，发现缺失步骤
- **参数对比**：比较两台仪器的配置参数，快速找出差异
- **电机运动精度分析**：通过光栅偏差数据分析定位精度问题

### 1.2 运行环境

| 项目 | 要求 |
|------|------|
| 操作系统 | Windows 10/11（64位） |
| Python 版本 | 3.8+（开发环境）|
| GUI 框架 | tkinter（Python 标准库） |
| 发布形式 | PyInstaller 打包的独立 EXE，无需安装 Python |

### 1.3 依赖项

```
pyserial >= 3.5     # 串口通信（serial_assistant.py 使用）
openpyxl >= 3.0     # 可选：从 .xlsx 文件加载动作名称
tkinter             # GUI（Python 标准库自带）
```

详见 `requirements.txt`。

## 3. 核心数据模型

所有数据模型使用 Python `@dataclass` 定义，位于 `fa120_log_parser.py` 第 50–134 行。

### 3.1 SampleInfo — 样本信息

```python
@dataclass
class SampleInfo:
    serial: str          # 4位样本序号，如 "0001"
    sample_id: str       # 完整样本ID，如 "202603110001"
    rack_pos: str        # 样本架位置，如 "0-0"
    test_count: int      # 检测项目数量
    cap_open: bool       # 是否开盖
    shake: bool          # 是否震荡
    dilution: int        # 稀释倍数（1/2/4/8...）
    sample_type: str     # 样本类型
    test_items: list     # 检测项目列表
    arrange_time: str    # 安排时间 HH:MM:SS
    mode: str            # 检测模式（一步法/单次稀释/多次稀释/两孔稀释）
    status: str          # 状态（测试完成/异常/未知）
    finish_time: str     # 完成时间 HH:MM:SS
    project_abbr: str    # 检测项目缩写
    concentration: str   # 浓度值
    measure_value: str   # 测量值
    missing_actions: list  # 缺少的一级动作码列表
```

**mode 取值规则：**  
根据解析到的动作的 `mode_char` 字段推断：
- `mode_char = B` → 两孔稀释  
- `mode_char = C` → 单次稀释  
- `mode_char = D` → 多次稀释  
- `mode_char = E` → 一步法  

### 3.2 Action — 动作记录

```python
@dataclass
class Action:
    sample_num: str      # 5位样本号，"00000"表示系统预设动作
    mode_char: str       # 模式字母（A-E）
    level1: str          # 一级动作码，如 "E00"、"F07"
    level2: str          # 二级动作/子步骤，如 "1.5"、"3.3"
    component: str       # 执行动作的电机/组件名
    start_pos: int       # 起始位置坐标（电机编码器值）
    end_pos: int         # 目标位置坐标
    start_time: str      # 动作开始时间 HH:MM:SS.mmm
    end_time: str        # 动作结束时间 HH:MM:SS.mmm
    theory_raster: int   # 理论光栅数（期望步数）
    actual_raster: int   # 实际光栅数（电机反馈）
    raster_deviation: int  # 偏差 = actual - theory（≥3 则表格标红）
    _action_key: str     # 内部键，用于关联 Start/Finish/MOTOR 日志行
```

**动作码命名规则：**  
格式为 `[模式字母][两位数字]`，例如：
- `E00` = E模式（一步法）第0步
- `F07` = F模式第7步
- 纯字母如 `ADP` 代表液路模块动作

### 3.3 Alarm — 报警记录

```python
@dataclass
class Alarm:
    time_str: str     # 报警时间 HH:MM:SS
    error_code: str   # 错误码，如 "G-INJ-012"
    sample_num: str   # 关联的样本序号
    action_code: str  # 关联的动作码
    content: str      # 错误描述
    detail: str       # 详细信息
```

解析时自动去重：同一秒内完全相同的报警只保留一条。

### 3.4 InstrumentInfo — 仪器信息

```python
@dataclass
class InstrumentInfo:
    device_serial: str           # 仪器序列号（SN）
    log_time: str                # 日志时间跨度
    user_program_version: str    # 用户程序版本
    request_count: int           # 请求总数
    detect_count: int            # 检测总数
    open_cap_count: int          # 开盖次数
    current_arrangement_count: int  # 当前批次样本数
    ccid: str                    # 4G 模块 CCID
    signal_strength: str         # 网络信号强度
    mid_version: str             # 主控板固件版本
    mcu1_version: str            # MCU1 固件版本
    mcu2_version: str            # MCU2 固件版本
    mcu3_version: str            # MCU3 固件版本
    temp_control_version: str    # 温控板固件版本
```

### 3.5 其他模型

- **SelfCheckRecord**（第 121 行）：自检记录，含时间戳和结果
- **UserActionRecord**（第 128 行）：用户操作记录，含操作类型和时间

## 4. 日志解析引擎（LogParser）

**位置：** `fa120_log_parser.py` 第 138–821 行  
**职责：** 将原始日志文本转化为结构化的 Python 对象

### 4.1 正则表达式设计

| 正则变量 | 匹配目标 | 提取内容 |
|---------|---------|---------|
| `RE_SAMPLE` | 样本请求行 | 样本ID、架位、检测数、稀释、开盖、震荡等 |
| `RE_START` | 电机动作开始行 | 时间、样本号、模式字母、动作码、组件名、坐标 |
| `RE_START_ADP` | ADP液路动作开始行 | 同上，但无组件名 |
| `RE_FINISH` | 电机动作结束行 | 时间、样本号、动作码 |
| `RE_FINISH_ADP` | ADP动作结束行 | 同上 |
| `RE_MOTOR` | 电机光栅信息行 | 理论光栅数、实际光栅数 |
| `RE_ALARM` | 报警/错误行 | 时间、错误码、样本号、动作码、描述 |
| `RE_TIME_PREFIX` | 提取时间前缀 | HH:MM:SS |
| `RE_THEORY` | 理论时间格式 | 动作码和毫秒数 `(1000)` |

### 4.2 核心属性

```python
self.samples: dict[str, SampleInfo]     # 键：4位序号 "0001"
self.actions: dict[str, list[Action]]   # 键：4位序号，值：该样本所有动作
self.system_actions: list[Action]       # 系统预设动作（sample_num="00000"）
self.alarms: list[Alarm]               # 全部报警
self.instrument_info: InstrumentInfo   # 仪器元数据
self.all_components: list[str]         # 出现过的所有电机/组件名（有序）
self._pending: dict[str, Action]       # 等待 Finish 行的半成品 Action
```

### 4.3 主要方法

#### `load_file(path: str)`（第 343–410 行）

完整解析流程入口：
1. 自动检测文件编码（UTF-8 → GBK → GB2312 → GB18030 依次尝试）
2. 逐行遍历，依次尝试所有正则匹配
3. 解析完成后调用后处理：推断模式、判断状态、检查完整性

#### `_parse_sample(line)`（第 678–697 行）

从样本请求行提取 SampleInfo，存入 `self.samples[serial]`。

#### `_parse_start(line)`（第 705–754 行）

解析动作开始行，创建 Action 对象：
- 若 `sample_num == "00000"` → 存入 `self.system_actions`
- 否则存入 `self.actions[serial]` 并存入 `self._pending[key]` 等待结束行

#### `_parse_finish(line)`（第 756–770 行）

解析动作结束行，通过 `_action_key` 找到对应的 pending Action，写入 `end_time`。

#### `_parse_motor(line)`（第 772–789 行）

解析光栅信息行，更新 pending Action 的 `theory_raster`、`actual_raster`、`raster_deviation`。

#### `_parse_alarm(line)`（第 791–821 行）

解析报警行，去重后加入 `self.alarms`。

#### `_check_action_completeness()`（第 644–676 行）

对每个样本检查动作完整性：
1. 以 0001 号样本为基准，学习标准序列
2. 对比其他样本的一级动作码与标准序列
3. 缺失动作记录在 `sample.missing_actions`

#### 辅助加载方法

| 方法 | 来源 | 格式说明 |
|------|------|---------|
| `load_theory_time(path)` | 文本文件 | `E00：取TIP(2500)` |
| `load_motor_names(path)` | 文本文件 | 每行一个组件名 |
| `load_action_names(path)` | 文本/XLSX | 多种格式，见代码注释 |

### 4.4 时间工具函数

```python
time_to_ms(t: str) -> float
    # "HH:MM:SS.mmm" → 毫秒数（浮点）

ms_to_time(ms_val: float) -> str
    # 毫秒数 → "HH:MM:SS.mmm"
```

### 5.2 TimelineCanvas — 时间轴画布

**位置：** 第 866–1149 行，继承 `tk.Frame`

**视觉元素：**
- **X轴**：时间刻度（HH:MM:SS），自动根据缩放级别调整间隔
- **Y轴**：电机/组件名称（固定在左侧，可拖动调整顺序）
- **彩色方块**：每个动作为一个矩形，颜色按一级动作码分组（8色循环）
- **红色竖线**：报警时刻标记
- **交替背景色**：按一级动作码分区着色，便于区分流程段

**交互功能：**
- 鼠标滚轮水平缩放（范围：0.02 ~ 5 像素/毫秒）
- 水平滚动条
- 点击报警标记 → 高亮对应的表格行

**关键方法：**

| 方法 | 功能 |
|------|------|
| `set_data(actions, alarms, components)` | 加载数据 |
| `_draw()` | 重绘整个画布 |
| `_draw_regions()` | 绘制按一级动作分区的背景色 |
| `_draw_time_axis()` | 绘制时间刻度（自适应间隔） |
| `scroll_to_time(t)` | 跳转到指定时间位置 |

### 5.3 TableView — 动作详情表格

**位置：** 第 1153–1475 行，继承 `tk.Frame`

**列结构：**

| 列名 | 类型 | 说明 |
|------|------|------|
| 动作码 | 固定 | 一级/二级动作码 |
| 动作名称 | 固定 | 来自 theory_data.json |
| 子步骤 | 固定 | level2 字段 |
| 组件 | 固定 | 电机/组件名 |
| 坐标 | 可选 | 起始/终止位置 |
| 开始时间 | 可选 | HH:MM:SS.mmm |
| 结束时间 | 可选 | HH:MM:SS.mmm |
| 耗时(ms) | 可选 | 实际执行时长 |
| 理论光栅 | 可选 | 期望步数 |
| 实际光栅 | 可选 | 实际步数 |
| 光栅偏差 | 可选 | 实际-理论，≥3 标红 |

**行着色规则：**
- **父行（一级动作）**：
  - 正常完成 → 浅蓝背景
  - 超时 → 红色文字 + 显示超时量
- **子行（子步骤）**：
  - 涉及报警 → 浅红背景
  - 光栅偏差 ≥ 3 → 红色文字

**交互功能：**
- 双击父行 → 展开/折叠子步骤
- 列可见性切换（通过右键菜单或顶部按钮）

**关键方法：**

| 方法 | 功能 |
|------|------|
| `_build_ordered_batches()` | 将连续的一级动作分组，处理交错动作 |
| `_repopulate()` | 根据数据重建整个树 |
| `highlight_action(code)` | 滚动并高亮指定动作码 |

## 6. 功能弹窗与设置

### 6.1 TheoryTimeDialog — 理论时间设置

**位置：** 第 1479–1617 行

用于编辑各动作码的理论参数，双击单元格即可修改：

| 可编辑字段 | 说明 |
|-----------|------|
| 动作描述 | 动作的完整说明文字 |
| 表格显示名 | 在 TableView 中显示的简短名称 |
| 理论时间(ms) | 该动作的期望执行时长（毫秒） |

保存格式：`E00：取TIP(2500)`  
持久化：修改后自动保存到 `theory_data.json`

### 6.2 MotorNamesDialog — 电机名称别名设置

**位置：** 第 1620–1715 行

将日志中解析出的原始电机名映射为界面显示名称。  
双击"显示名"列即可编辑，保存到 `motor_aliases.json`。

**6个必需组件（不可删除）：**
- 进卡Y、退卡Y、样本检测Y、进样本架Y、退样本架Y、摆渡车X

### 6.3 StandardFlowDialog — 标准流程管理

**位置：** 第 2330–2467 行

**功能：**
- 加载参考日志，自动从 0001 号样本学习标准动作序列
- 按检测模式分别存储：两孔稀释 / 单次稀释 / 多次稀释 / 一步法
- 支持查看和清空已存储的序列
- 序列保存到 `standard_sequences.json`

**用途：** 解析其他样本时，软件以此为基准检测缺失步骤。

### 6.4 ParamsView — 参数对比视图

**位置：** 第 1720–2220 行

**功能：** 加载两个仪器的 JSON 参数文件进行对比。

**参数树结构：**
```
系统信息（SN、固件版本）
├── 主板参数
├── ADP 参数
├── 电机参数（按电机命名）
├── 坐标参数（各电机的定位点 + 动作参数）
└── 温控参数
```

**差异高亮规则：**
- 红色背景：两份文件中值不同
- 黄色背景：某一份文件中缺少该参数

**附加功能：**
- 仅显示差异项（过滤开关）
- 全文搜索参数名
- 双击可编辑参数名称或添加备注
- 参数名和备注持久化到 `param_action_names.json`

## 8. 数据流程

```
原始日志文件（.txt / .log）
         │
         ▼
┌─────────────────────────────────────┐
│         编码自动检测                 │
```

### 10.5 已知问题与注意事项

**1. tkinter trace_variable 废弃警告**

`app.log` 中会出现：
```
DeprecationWarning: trace_variable() is deprecated; use trace_add() instead.
```
位置：`fa120_log_parser.py` 第 2785 行（ParamsView 搜索变量）。  
建议将 `.trace_variable('w', ...)` 改为 `.trace_add('write', ...)`。

**2. 混合编码日志**

编码检测按序尝试，但极少数混合编码文件可能解析异常。  
遇到乱码时，可在 `load_file()` 开头手动指定 encoding 参数调试。

**3. 大文件性能**

对于 10MB 以上的日志，逐行正则解析耗时较长。  
代码中已有 `multiprocessing` 和 `concurrent.futures` 的 import，可考虑将解析放入子进程并加进度条。

**4. `_pending` 字典内存占用**

若日志缺少 Finish 行（仪器异常关机等），`_pending` 中的 Action 对象不会被释放。  
在 `load_file()` 结束后可加清空逻辑。

### 10.6 串口助手工具

`serial_assistant.py` 是独立的串口调试工具，可单独运行：

```bash
python serial_assistant.py
```

功能包括：
- 配置串口（波特率、数据位、奇偶校验、停止位、编码）
- 实时收发数据，支持 HEX/文本显示
- 定时自动发送
- 收发字节计数

与主解析程序完全独立，可按需维护。
